### 3.2 前后端职责划分

前端职责：

1. 展示页面
2. 收集客服输入
3. 显示推荐结果和投诉结果
4. 提示错误信息
5. 发起接口请求
6. 管理当前浏览器端展示状态

后端职责：

1. 执行业务规则
2. 调用 AI 服务
3. 管理会话上下文
4. 管理房间与表单逻辑
5. 读写 Excel
6. 控制保存一致性
7. 返回标准化响应

### 3.3 解耦原则

为了确保模块可独立测试，所有模块按以下原则设计：

1. 模块只通过明确接口交互，不直接依赖彼此内部实现。
2. AI 调用必须封装在独立服务中，其他模块不得直接访问模型接口。
3. Excel 操作必须集中在仓储层，业务模块不得直接操作文件。
4. 规则判断必须集中在规则层，页面和 AI 不直接做最终业务判定。
5. 每个模块都应支持依赖替身注入，以便单元测试。

## 4. 后端模块设计

后端采用 `FastAPI + Service + Repository + Rules + Schemas` 分层设计。

### 4.1 API 路由模块

路径建议：

1. `backend/app/api/conversation.py`
2. `backend/app/api/rooms.py`
3. `backend/app/api/forms.py`
4. `backend/app/api/complaints.py`
5. `backend/app/api/files.py`

职责：

1. 接收 HTTP 请求
2. 参数解析
3. 调用对应服务
4. 返回标准响应

不负责：

1. 业务规则判断
2. Excel 文件操作
3. AI Prompt 组织细节

可单测点：

1. 参数校验是否生效
2. 错误码是否正确
3. 响应结构是否符合约定

### 4.2 会话上下文模块

建议文件：

1. `backend/app/services/conversation_context_service.py`

职责：

1. 保存当前会话的已提取信息
2. 保存当前会话的补充状态
3. 保存当前会话最近一次推荐结果
4. 执行“新对话清空”

输入：

1. `session_id`
2. 当前用户输入文本
3. 已抽取字段

输出：

1. 当前完整上下文对象

依赖：

1. 内存存储适配器

说明：

1. 当前阶段可先用进程内存。
2. 后续如需多实例部署，应替换为 Redis 等外部存储。

可单测点：

1. 上下文是否正确累积
2. 补充输入是否能续接旧信息
3. 新对话是否彻底清空

### 4.3 AI 网关模块

建议文件：

1. `backend/app/services/ai_gateway.py`

职责：

1. 统一调用大模型接口
2. 管理 Prompt 模板
3. 封装模型返回值
4. 处理超时、空结果、结构异常

输入：

1. Prompt 名称
2. Prompt 参数
3. 期望输出结构

输出：

1. 标准化 AI 结果对象

不负责：

1. 最终业务决策
2. 直接修改表单或房态

可单测点：

1. Prompt 参数拼装
2. 返回结构转换
3. 异常处理逻辑

测试策略：

1. 单测时必须使用 Mock，不直接访问真实模型。

### 4.4 意图识别模块

建议文件：

1. `backend/app/services/intent_service.py`

职责：

1. 判断输入是普通答疑还是订房需求

输入：

输出：

1. `qa`

依赖：

可单测点：

1. `BookingDraft`

3. 匹配人数容量
4. 预算约束判断
5. 方案排序

1. 对已选定方案生成推荐理由和自然语言文案

可单测点：

1. 单方案场景
2. 多方案场景

## 6. 模块接口设计

以下为建议接口设计，按字段级别定义。

### 6.1 上传 Excel

`POST /api/files/upload-excel`

请求：

1. `file`

响应成功：

```json
{
  "success": true,
  "message": "Excel 上传成功",
  "data": {
    "file_name": "hotel_data.xlsx"
  }
}
```

响应失败：

```json
{
  "success": false,
  "error_code": "INVALID_EXCEL_FILE",
  "message": "上传失败：文件不是有效的 Excel 格式"
}
```

### 6.2 新对话

`POST /api/conversation/reset`

请求：

```json
{
  "session_id": "abc123"
}
```

响应：

```json
{
  "success": true,
  "message": "新对话已开始"
}
```

### 6.3 首页意图分析

`POST /api/conversation/analyze`


响应三：订房方案

```json
{
  "success": true,

    "intent": "booking",
```

## 7. 数据结构设计

建议统一放在：

1. `backend/app/schemas/`
2. `backend/app/models/`

### 7.1 Room

```python
class Room:
    room_number: str
    floor: int
    room_name: str
    capacity_type: str
    room_category: str
    special_type: str | None
    price_per_day: float
    status: str
    image_url: str
```

### 7.2 BookingDraft

```python
class BookingDraft:
    guest_count: int | None
```

## 8. 测试设计要求

你要求模块尽量独立并能独立测试，因此每个模块都必须满足以下要求。

### 8.1 测试原则

1. 每个服务模块必须能在不启动前端的情况下测试。
2. 每个规则模块必须能纯函数化测试。
3. 每个仓储模块必须能使用临时 Excel 文件测试。
4. AI 模块测试必须使用 Mock。
5. 接口测试必须覆盖成功、失败、边界三类场景。

### 8.2 推荐的测试分层

1. 规则单元测试
2. 服务单元测试
3. 仓储集成测试
4. API 接口测试

### 8.3 每类模块的依赖替身要求

1. AI 网关：使用假响应替身
2. Excel 仓储：使用临时测试文件
3. 会话上下文：使用内存假仓储
4. 文件上传：使用测试上传对象

### 8.4 关键失败场景必须测试

1. 未上传 Excel 就保存表单
2. Excel 文件损坏
3. Excel 被占用
4. 推荐结果中包含非空房时被规则拦截
5. 待完成表单修改其他字段被拦截
6. 新对话后上下文未清空
7. AI 返回结构缺字段

## 9. Docker 设计

你要求做好 Docker，因此后端必须按 Docker Space 标准设计。

### 9.1 后端 Dockerfile

建议位置：

1. `backend/Dockerfile`

要求：

1. 使用精简 Python 基础镜像
2. 明确工作目录
3. 只复制后端运行需要的文件
4. 不把 `.env` 复制进镜像
5. 启动命令固定为 FastAPI 服务

建议逻辑：

1. `FROM python:3.11-slim`
2. 安装依赖
3. 复制 `backend/`
4. 暴露 `7860` 或 Hugging Face 指定端口
5. 启动 `uvicorn`

### 9.2 docker-compose

建议根目录提供：

1. `docker-compose.yml`

作用：

1. 本地联调前端和后端
2. 统一管理环境变量

建议服务：

1. `backend`
2. `frontend`

### 9.3 Docker 安全要求

1. 镜像中不得包含真实密钥
2. 不得复制本地 Excel 真实业务文件
3. 不得复制测试缓存和日志
4. 必须使用 `.dockerignore`

## 10. GitHub 上传与防泄漏设计

这是必须重点控制的部分。

### 10.1 禁止提交的内容

## 13. 模块独立开发建议

为了方便后续独立开发和独立测试，建议按以下顺序推进：

1. 先实现 Schemas、Rules、Excel Repository
2. 再实现 Room Service、Form Service
3. 再实现 Conversation Context、Intent、Booking Extract、Recommendation

3. AI 模块可后接，不会阻塞基础能力
4. 前端联调时后端边界已清晰
