# 酒店客服自动化系统需求文档

## 1. 文档概述

本文档用于定义“酒店客服自动化系统”的正式需求范围，作为后续产品设计、页面设计、技术实现和测试验收的依据。

本系统定位为一个演示版酒店客服工作台，面向客服人员使用，核心目标是提升订房推荐、表单管理、房态管理、知识问答和投诉处理的效率。

## 2. 项目目标

系统需实现以下目标：

1. 帮助客服快速处理客户咨询、订房推荐和投诉场景。
2. 借助大模型自动识别客户输入内容，减少人工整理信息的时间。
3. 仅基于当前可售空房进行推荐，避免推荐已住或已预订房间。
4. 通过表单与 Excel 联动，实现订单和房态信息留存。
5. 输出规范、正式、可直接复制使用的客服回复内容。

## 3. 使用角色

当前系统仅考虑一种角色：

1. 客服人员

客服人员可执行以下操作：

1. 在首页输入客户原话或客服整理后的内容。
2. 发起新对话或切换到新对话。
3. 查看大模型返回的问答结果或订房方案。
4. 选择推荐方案并自动生成表单。
5. 手动补充联系人、手机号、性别。
6. 查看和处理待完成表单。
7. 查询房间信息并手动调整房态。
8. 上传或替换当前使用的 Excel 文件。
9. 输入投诉内容并获取处理建议与上报摘要。

## 4. 业务边界

本期系统为演示版，按以下范围实施：

1. 酒店共有 32 个房间。
2. 房价为固定单价，不同房型价格不同。
3. Excel 是当前唯一的数据落地载体。
4. 知识库内容采用演示数据，但应尽量符合正常酒店业务。
5. 入住日期按演示规则处理为“系统当前电脑日期”。

本期暂不支持：

1. 未来日期预订。
2. 动态房价。
3. 多酒店管理。
4. 多角色权限管理。

## 5. 基础业务定义

### 5.1 房态定义

系统仅保留以下三种房态：

1. 空房
2. 已预订
3. 已住

说明：

1. “已出租”与“已住”视为同一含义，不再单独作为独立状态存在。

### 5.2 日期定义

系统只保留一个日期字段：

1. 入住日期

字段规则：

1. 入住日期等于系统当前电脑时间对应的日期。
2. 入住日期由系统自动填入。
3. 不从客户话术中识别未来入住日期。

### 5.3 人数类型定义

人数类型包括：

1. 个人
2. 多人
3. 企业团建
4. 情侣

识别规则：

1. 如果大模型明确识别到“企业团建”，则使用“企业团建”。
2. 如果大模型明确识别到“情侣”，则使用“情侣”。
3. 如果是单人入住，则使用“个人”。
4. 如果是多人入住，且未明确识别为“企业团建”或“情侣”，则默认使用“多人”。

### 5.4 金额定义

表单中的金额字段表示：

1. 本次订单总金额

## 6. 房间数据规则

### 6.1 楼层与房号

酒店共 6 层：

1. 1 层为大厅，不住客。
2. 2 层房号为 `201-207`。
3. 3 层房号为 `301-307`。
4. 4 层房号为 `401-407`。
5. 5 层房号为 `501-507`。
6. 6 层房号为 `601-604`。

合计 32 个房间。

### 6.2 房型结构

房型和筛选维度定义如下：

1. `单人间 / 多人间` 为入住容量维度。
2. `普通房 / 特殊房` 为房型层级维度。
3. `情侣套房 / 文创套房` 为特殊房的具体类型。

当前房型范围如下：

1. 单人间
2. 多人间
3. 普通房
4. 特殊房
5. 情侣套房
6. 文创套房

说明：

1. 每个房间固定绑定一个房型和一个固定价格。
2. 同一个房间的房型和价格不会变化。

### 6.3 初始房态

系统初始化时需生成 32 个房间数据，并按演示要求处理初始房态：

1. 随机设置 7 个房间为“已住”。
2. 其余房间默认为“空房”。

## 7. Excel 数据规则

### 7.1 上传规则

系统必须依赖 Excel 文件保存数据。

规则如下：

1. 首页必须提供 Excel 上传入口或明显提示。
2. 用户可以随时替换新的 Excel 文件。
3. 当前系统产生的数据只写入当前上传的 Excel 文件。
4. 如果未上传 Excel，则所有需要保存数据的操作必须禁止，并明确提示用户先上传 Excel。

### 7.2 需禁止的操作

在未上传 Excel 时，至少以下操作必须禁止：

1. 新增表单
2. 修改表单
3. 修改房态

### 7.3 Excel 模板设计原则

Excel 模板由系统方案统一设计，需至少支持以下信息记录：

1. 表单唯一标识
2. 联系人
3. 性别
4. 手机号
5. 总金额
6. 人数
7. 人数类型
8. 入住日期
9. 居住天数
10. 房间号
11. 订单状态
12. 房态同步结果
13. 创建时间
14. 更新时间

### 7.4 更新规则

1. 已完成表单和待完成表单都要写入 Excel。
2. 待完成表单后续改为已完成时，应更新原记录，不新增新记录。
3. 房态变更后，Excel 中对应记录也应同步更新。

## 8. 页面与功能需求

### 8.1 首页

首页是客服主要工作入口，应突出输入客户话语的核心能力。

首页需包含：

1. 客户话语输入区
2. 发送或分析入口
3. 新对话入口
4. 新增表单入口
5. 房间查询入口
6. 投诉处理入口
7. Excel 上传入口或提示区

前端设计要求：

1. 整体风格偏潮流、简约、高效。
2. 首页重点突出输入区域。
3. 导航可以采用按钮或侧边栏。
4. 页面风格不能粗糙。

### 8.2 新对话功能

系统必须支持“新对话”能力。

规则如下：

1. 默认情况下，当前对话保留上下文。
2. 当系统提示信息不足时，客服只需继续补充缺失信息，无需重开流程。
3. 客服点击“新对话”后，上一轮上下文必须彻底清空。
4. 新对话不能继承上一位客户的任何需求信息。

### 8.3 表单字段

正式表单字段如下：

1. 联系人
2. 性别
3. 手机号
4. 总金额
5. 人数
6. 人数类型
7. 入住日期
8. 居住天数
9. 房间号
10. 订单状态

字段规则：

1. 入住日期由系统自动填入。
2. 房间号支持单个或多个房间。
3. 房间号必须明确展示具体编号。
4. 订单状态包含“已完成”和“待完成”。
5. 订单状态默认值为“已完成”。
6. 联系人、手机号、性别由客服手动填写。

### 8.4 新增表单流程

点击“新增表单”后，弹出两个选项：

1. 待完成表单
2. 新增表单

#### 8.4.1 待完成表单流程

1. 如果没有待完成表单，则显示“所有表单已完成”提示框。
2. 提示框支持两种关闭方式：
   1. 5 秒后自动返回首页。
   2. 点击关闭按钮返回首页。
3. 如果存在待完成表单，则横向展示所有待完成表单。
4. 点击某个待完成表单后，进入该表单详情页。
5. 客服只能将其状态从“待完成”改为“已完成”。
6. 客服不能在该步骤中修改房间、金额、居住天数、人数等其他业务信息。
7. 若改完后已无待完成表单，则直接返回首页。
8. 若改完后仍有待完成表单，则返回待完成表单列表页，继续处理。
9. 在整个新增表单流程中，点击空白区域可返回上一步，最多返回到首页。

#### 8.4.2 新增表单流程

1. 客服可手动新建表单。
2. 也可由大模型推荐方案转为表单。

### 8.5 房间查询

系统应支持房间列表展示与筛选。

每个房间至少展示以下信息：

1. 房间图片
2. 房间价格
3. 房型名称
4. 房态
5. 房间号

筛选条件包括：

1. 单人间 / 多人间 / 全部
2. 空房 / 已住 / 已预订 / 全部
3. 普通房 / 特殊房 / 全部

### 8.6 房态维护

客服可手动维护房态。

规则如下：

1. 客服可以将“已住”房间调整为空房。
2. 客服可以将“已预订”房间调整为空房。
3. 房态调整后，应同步更新到当前 Excel。
4. 被客服手动改为空房的房间，后续可以重新参与大模型推荐。

## 9. 首页智能处理逻辑

### 9.1 意图判断

首页接收客户话语后，大模型需先判断当前属于哪类任务：

1. 普通答疑
2. 订房需求

### 9.2 普通答疑

如果判断为普通答疑，则调用知识库回答。

知识库至少应覆盖以下常见酒店问题：

1. 退房时间
2. 保洁时间
3. 早餐价格
4. 早餐时间
5. 能否加床
6. 是否可带宠物
7. 停车规则
8. 发票说明
9. Wi-Fi 使用说明
10. 入住与押金说明

输出要求：

1. 语气正式、礼貌。
2. 回答内容简洁清楚。
3. 支持客服直接复制给用户。

### 9.3 订房需求识别

如果判断为订房需求，则大模型需从输入中抽取结构化信息。

可抽取字段包括：

1. 人数
2. 房间数
3. 预算
4. 居住天数
5. 人数类型
6. 其他补充偏好

说明：

1. 入住日期不从客户话术识别，而是系统自动取当前日期。

输入内容可能表现为：

1. 单行短句
2. 多行聊天记录
3. 口语化描述
4. 非标准表达

示例：

1. `7人，2房，1000r，3天`
2. `我们有3个人，打算住1天，开2个房间，怎么便宜怎么来`

### 9.4 信息不足处理

如果识别为订房需求但信息不足，系统需：

1. 明确指出缺失了哪些信息。
2. 保留当前对话上下文。
3. 等待客服继续补充信息。
4. 在客服补充后继续原流程，不重新开始。

### 9.5 订房方案生成

当信息足够后，系统需生成订房方案。

规则如下：

1. 仅在空房中检索可推荐房间。
2. 已住和已预订房间必须在检索阶段直接跳过。
3. 结合人数、房间数、预算、居住天数、房型和固定价格计算方案。
4. 生成 1 到 3 个可行方案。
5. 如果只有 1 个方案，则只输出 1 个。
6. 每个方案必须提供推荐理由。
7. 输出内容应便于客服直接复制。
8. 所有回复必须正式、规范，不得冒犯他人。

## 10. 推荐方案转表单

每个推荐方案后需提供可选圆圈。

处理规则如下：

1. 客服点击圆圈后，圆圈变为对勾。
2. 系统自动创建新表单。
3. 系统自动填入已识别出的业务信息。
4. 客服手动补充联系人、手机号、性别。
5. 表单支持记录多个房间号。

## 11. 表单状态与房态联动

表单状态直接决定房态变化。

联动规则如下：

1. 如果表单创建时状态为“已完成”，则表单中的房间立即切换为“已住”。
2. 如果表单创建时状态为“待完成”，则表单中的房间立即切换为“已预订”。
3. 待完成表单后续被修改为“已完成”时，对应房间再从“已预订”切换为“已住”。
4. 房态联动结果必须同步写入当前 Excel。

## 12. 投诉处理需求

### 12.1 投诉处理入口

系统需提供独立的投诉处理页面或模式，交互方式与首页相近，核心仍然是文本输入。

### 12.2 投诉分类

投诉需先识别类型，类型包括：

1. 卫生
2. 服务态度
3. 设施故障
4. 噪音
5. 退款

### 12.3 严重程度划分

投诉严重程度分为：

1. 轻度
2. 中度
3. 重度

演示版判断规则如下：

1. 轻度投诉必须是确实因酒店问题影响了客户，但影响范围较小、可当场补救、可直接补偿处理的情况。
2. 中度投诉为影响较明显、客服不适合自行拍板、需要上报处理的情况。
3. 重度投诉为影响严重、争议较大、风险较高或明显需要管理层介入的情况。

### 12.4 投诉输出规则

1. 轻度投诉：输出安抚文案、解决方式、赔偿措施。
2. 中度投诉：输出安抚文案、上报上级提示、上报摘要。
3. 重度投诉：输出安抚文案、上报上级提示、上报摘要。
4. 中度和重度不输出具体赔偿措施和具体解决承诺。
5. 赔偿措施可采用合理演示方案，如赠送餐食、赠送礼品、适度减免等。
6. 所有措辞必须正式、规范、克制。

## 13. 工作流设计

### 13.1 订房工作流

1. 客服输入客户原话
2. 大模型判断是否属于订房需求
3. 若是，则抽取人数、房间数、预算、居住天数、人数类型等信息
4. 若信息不足，则提示缺失字段并等待补充
5. 信息足够后，在空房中检索可用房间
6. 根据房型规则和固定价格生成 1 到 3 个方案
7. 输出推荐理由与规范回复文案
8. 客服选择方案后自动生成表单
9. 根据表单状态同步房态和 Excel

### 13.2 普通答疑工作流

1. 客服输入客户问题
2. 大模型判断为普通答疑
3. 调用知识库查询答案
4. 输出规范回复内容

### 13.3 投诉处理工作流

1. 客服输入投诉内容
2. 大模型识别投诉类型
3. 判断严重程度
4. 轻度投诉输出安抚文案、解决方式、赔偿措施
5. 中重度投诉输出安抚文案、上报提示、上报摘要

## 14. System Prompt 节点要求

系统需为以下节点分别配置独立的 system prompt：

1. 首页意图识别
2. 普通答疑
3. 订房需求信息抽取
4. 缺失信息追问
5. 订房方案生成
6. 客服回复文案生成
7. 投诉类型识别
8. 投诉轻重判断
9. 投诉安抚文案与上报摘要生成

统一要求：

1. 输出必须规范、稳定、可复用。
2. 不得输出冒犯性内容。
3. 不得推荐已住或已预订房间。
4. 必须优先遵守业务规则，再组织自然语言表达。

## 15. 风险与后续建议

### 15.1 API Key 安全

后续实现时，不得把真实 API Key 写入文档、前端代码或仓库。

建议：

1. 使用环境变量或 `.env` 文件管理密钥。
2. 如已有明文暴露，应立即更换。

### 15.2 Excel 方案适用范围

Excel 适合演示版，但不适合长期生产使用。

后续若系统真实投入使用，建议：

1. 改为数据库存储。
2. Excel 保留为导入导出工具。

### 15.3 日期规则限制

当前演示规则中，入住日期固定等于系统当天日期，因此暂不支持提前预订未来入住场景。

后续若要支持真实预订，应拆分为两个字段：

1. 当前日期
2. 入住日期

## 16. 验收重点

系统完成后，至少应满足以下验收标准：

1. 未上传 Excel 时，新增表单、修改表单、修改房态均被阻止。
2. 房间初始化后共有 32 个房间，且房号固定正确。
3. 首页可识别普通答疑与订房需求。
4. 订房推荐不会出现已住或已预订房间。
5. 信息不足时可连续补充，不丢失上下文。
6. 点击新对话后，上下文彻底清空。
7. 方案转表单后，可自动填充业务信息。
8. 待完成表单改为已完成后，房态同步正确，Excel 记录更新而不是新增。
9. 投诉处理可输出对应类型、严重程度、安抚文案和上报摘要。
10. 所有输出文案符合正式、规范、礼貌的要求。
